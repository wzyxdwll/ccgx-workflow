# Migration Guide: CCG v4.1 → v4.2

**Released**: 2026-05-04
**Phase scope**: P21 (multi-model routing SSoT) → P23 (quality tier dogfood + docs)

> v4.2 is the **multi-model collaboration depth** release. The headline feature is the **`--quality=fast|triple|debate` tier flag**, which scales main-thread × subagent collaboration from "single phase-runner" (v4.1 default) to a **Plan-Critic-Verify three-stage orchestration** (v4.2 default) and an optional **multi-round debate** mode for Critical phases.
>
> ⚠️ **Default behaviour change**: `/ccg:autonomous` now defaults to `triple` tier (4 waves per phase) instead of v4.1's single-wave phase-runner. To reproduce v4.1 behaviour, pass `--quality=fast`.

---

## TL;DR

- **New `--quality=<tier>` flag** for `/ccg:autonomous` with three tiers: `fast` / `triple` / `debate`
- **New phase frontmatter `Quality:` field** for per-phase override
- **Default tier changed to `triple`** — v4.1 single-wave behaviour available via `--quality=fast`
- **Multi-model routing SSoT** (`src/utils/multi-model-routing.ts`) consolidates Layer / Model / PluginAvailability across 4 routers
- **Auto-degradation** when plugins missing: `debate→triple→fast` on plugin loss
- **No template removals**, no command palette changes
- **Real plugin cold-start validation pending** — user-side after release; this guide includes a 5-step checklist

## New: `--quality=<tier>` flag

`/ccg:autonomous` now accepts `--quality=fast|triple|debate` to scale orchestration depth per run. Default is `triple`.

```bash
# Reproduce v4.1 behaviour (single phase-runner wave)
/ccg:autonomous --quality=fast

# Default v4.2 (Plan → Critic → Impl → Verify, 4 waves)
/ccg:autonomous

# Multi-round debate (Plan → 3 debate rounds → Critic → Impl → Verify, 7 waves)
/ccg:autonomous --quality=debate
```

### Per-tier orchestration shape

| Tier | Waves | Spawns (backend) | Spawns (fullstack) | Wall-clock vs v4.1 fast | Use case |
|------|-------|------------------|--------------------|--------------------------|----------|
| `fast`   | 2 (impl → verify)                                              | 2  | 2  | +30%        | Hotfix / simple phase |
| `triple` (default) | 4 (plan → critic → impl → verify)                           | 8  | 8  | +60-90%     | Regular development |
| `debate` | 7 (plan → 3× debate-round → critic → impl → verify)            | 11 | 14 | +100-150%   | Critical decisions (architecture, breaking, data-loss) |

Spawn breakdown for `triple`: 3 plan (codex + gemini + claude lateral diversity) + 2 critic (assumptions-analyzer + nyquist-auditor) + 1 phase-runner impl + 2 cross-vendor verify.

`debate` debate-round count is **hard-capped at 3** to bound wall-clock.

## New: phase-level `Quality:` frontmatter override

Individual phases in `.ccg/roadmap.md` can override the global `--quality` flag with a `Quality:` frontmatter field:

```yaml
---
ID: phase-7-auth-rewrite
Type: backend
Critical: true
Quality: debate          # ← per-phase override; wins over CLI flag
Goal: replace bespoke JWT with OAuth2 PKCE
---
```

**Resolution priority** (highest first):
1. Phase frontmatter `Quality:` field
2. CLI `--quality=<tier>` flag
3. Default `triple`

This lets you keep most phases on `triple` while elevating only the breaking ones to `debate`, without polluting CI invocation with global flags.

## Default behaviour change: `triple` instead of single-wave

**Before (v4.1)**:
```bash
/ccg:autonomous              # single phase-runner wave (impl + handoff)
```

**After (v4.2)**:
```bash
/ccg:autonomous              # default triple (plan + critic + impl + verify, 4 waves)
/ccg:autonomous --quality=fast   # opt-in v4.1 single-wave behaviour
```

**Action required**:
- If you have CI scripts that depend on per-phase wall-clock fitting under a tight budget, pass `--quality=fast` explicitly.
- If you're comfortable with +60-90% wall-clock for materially better quality (cross-vendor verify catches race / data-drift bugs that single-wave misses), keep the default.

## Auto-degradation when plugins missing

The router auto-degrades when codex / gemini plugins are unavailable:

| Requested tier | Plugin state | Effective tier | Why |
|----------------|--------------|----------------|-----|
| `debate` | both missing | `fast` | debate loses lateral diversity entirely |
| `debate` | one missing | `triple` | debate needs both plugins for adversarial pairing |
| `triple` | both missing | `fast` | plan/verify diversity collapsed |
| `triple` | one missing | `triple` (wave-level fallback) | one plugin slot replaced by `general-purpose` + corresponding CCG prompt |
| `fast`   | both missing | `fast` (verify-only fallback) | verify wave goes general-purpose with `claude/reviewer.md` prompt |

Degradation is announced in the orchestrator banner; phase still completes.

## Internal architecture: multi-model routing SSoT

Four router modules (specialist-router / challenger-orchestrator / debate-orchestrator / phase-runner) historically each defined their own `Layer` / `Model` / `PluginAvailability` types. v4.2 consolidates these into `src/utils/multi-model-routing.ts` (SSoT), and the 4 routers re-export from there.

**Action required**: none for end users. Library consumers of `@ccg-workflow/index` continue to import the same names.

---

## Critical safety constraint: no real plugin spawn from subagent

`phase-runner` and other CCG subagents **cannot spawn `codex:codex-rescue` / `gemini:gemini-rescue` plugin Agents** at the engine layer (validated in v4.0.1 commit `a7cdffd`). All plugin spawns happen from the **main thread** only. The Plan-Critic-Verify orchestration in v4.2 is therefore a **main-thread-driven** wave plan, not a nested subagent spawn graph.

This constraint is unchanged from v4.0.1; v4.2 builds on top of it.

## Validation checklist post-upgrade

This 5-step checklist exercises every degradation path and the happy-path triple flow. **Recommended after every upgrade and especially after v4.2.0 since real cold-start plugin behaviour was not exercised in CI dogfood** (see `.claude/team-plan/phase-23-quality-tier-dogfood-report.md` §4 for the latent bug surface).

```bash
# Step 1. Install v4.2.0
npx ccg-workflow@latest init

# Step 2. (optional) prune leftovers
npx ccg-workflow init --sync

# Step 3. Plugins NOT installed: verify graceful degradation
mkdir /tmp/ccg-v42-coldstart-no-plugins && cd /tmp/ccg-v42-coldstart-no-plugins
git init
mkdir -p .ccg && cat > .ccg/roadmap.md <<'EOF'
# Test Roadmap

## phase-1
- ID: phase-1-trivial
- Type: docs
- Quality: triple
- Goal: write a hello-world README
EOF
/ccg:autonomous
# Expect: banner "triple → fast: both plugins unavailable"
# Expect: phase completes via general-purpose fallback (claude reviewer prompt)

# Step 4. Only ONE plugin (e.g. gemini): verify partial degradation
# Install gemini plugin only, then:
/ccg:autonomous --quality=debate
# Expect: banner "debate → triple: one plugin unavailable"
# Expect: 4-wave triple plan with codex slot replaced by general-purpose

# Step 5. BOTH plugins installed: real triple wave + token / wall-clock budget
# Install both codex and gemini plugins, run 1 real phase:
/ccg:autonomous
# Note start ctx% via /context
# After phase completes, note end ctx%
# Expect: ctx delta ≤ +5% per single-phase triple run.
# If consistently > +5%, file an issue with token snapshots.
```

If any step fails, file at https://github.com/wangzy-ai/ccg-workflow/issues with the banner output + the phase frontmatter.

## Known unverified items (latent bugs to watch)

The following were **not validated** in CI dogfood (engine-layer constraints prevent real plugin spawn from subagent context); user cold-start is the first real exercise:

1. **codex/gemini real summary format vs P21 parser assumptions** — `parseChallengerSummary` assumes `STATUS:/FINDINGS:/NOTES:` schema. Real plugin output may add markdown headings / use yaml-style fields / mix Chinese-English severity tags. Robust parser fallback (`parseFindings` empty array on exception) should hold but watch SUMMARY.md `raw` fields for `status: error` clusters.
2. **Plugin "completely missing" path** — degradation banner / general-purpose fallback only ran in synthetic test. First real "no plugins installed" cold-start is the first end-to-end test of the fallback prompt-loading path.
3. **Main-thread token budget vs design estimate** — table above shows +60-90% wall-clock for triple, +100-150% for debate. These are extrapolations from v4.1 +1%/phase drift, not measured in real triple/debate dogfood.
4. **race instance real catch** — verify wave is *designed* to catch races / data drift; no real race was injected in P22 / P23 tests. Inject one deliberately to validate.
5. **Phase frontmatter `Quality:` field end-to-end parsing** — `quality-router` consumes the field; the autonomous command template's frontmatter parser was not updated in P23. If the parser layer drops the field, phase-override is silently inactive.

See `.claude/team-plan/phase-23-quality-tier-dogfood-report.md` §4 for full latent bug analysis.
