# Migration Guide: CCG v4.4 → v4.5

**Released**: 2026-05-06
**Phase scope**: P1a (Bash subprocess MVP) → P3 (release docs)

> v4.5 is the **OS-process isolation** release. It treats the v4.4.x main-process RSS leak (uni-iam workload measured at 23GB → v4.5 design target < 8GB) by replacing in-process sidechain `Agent(subagent_type="phase-runner")` with `Bash(claude -p --agent ccg/phase-runner)` OS-level subprocess.

---

## TL;DR

- **No breaking changes** for end users on the command palette
- **Default behaviour change (small)**: nested G-plan opt-in is **off** — `--nested=off` is **100% equivalent** to v4.5 v1 baseline (commit `285b2ac`). Pass `--nested=on` to enable.
- **New flags**: `/ccg:autonomous --nested=on|off`, `/ccg:status --tail <job-id>`, `/ccg:status --cancel <phase-id>`
- **New infrastructure**: `templates/scripts/ccg-phase-runner-launcher.mjs` (process supervisor wrapper), `src/utils/process-tree.ts` (kill-tree + reconciler), `src/utils/broker-log.ts` (tx_id correlation)
- **`phase-runner.md` engine constraint section removed**: v4.0.1 "⚠️ 引擎层硬约束" section deleted — CLI subprocess mode unlocks Agent tool natively (T9 PoC verified)
- **Release entry criteria pending**: uni-iam project 5+ phase autonomous run with claude.exe RSS < 8GB throughout (user must validate)
- **Decision gates G2 (memory) + G3 (broker tx_id) PASS** in CI; cross-platform broker.log validation deferred to user dogfood

## Why v4.5

v4.4.x dogfood on the uni-iam project measured `claude.exe` RSS climbing to **23GB** during a 10+ phase autonomous run. Root cause: every `Agent(subagent_type="phase-runner")` spawn ran in-process as a sidechain — Node.js heap retention plus accumulated tool-call state could not be released even after sidechain exit.

v4.5 solves this at the OS layer:

```
v4.4 (in-process sidechain, leak-prone):
  claude.exe (orchestrator)
  ├─ sidechain-1 (phase-runner)        ← retained heap
  ├─ sidechain-2 (phase-runner)        ← retained heap
  └─ sidechain-N (phase-runner)        ← retained heap
  → cumulative RSS 23GB

v4.5 (Bash subprocess, OS-isolated):
  claude.exe (orchestrator, lean)       ← <1GB
  └─ Bash(claude -p --agent phase-runner)
       └─ CLI subprocess (phase work)   ← exits → memory reclaimed by OS
            └─ [opt-in] plugin process  ← --nested=on
  → design target main RSS <8GB throughout
```

When the CLI subprocess exits, the OS reclaims its full address space — no Node-level heap retention possible.

## What's new

### 1. Phase 1 (P1a, commit `e1f0fab`): Bash subprocess MVP

`phase-runner` spawn switched from `Agent(...)` to `Bash(claude -p --agent ccg/phase-runner ...)`.

New helper `buildPhaseRunnerBashCommand(phase, brief, jobId)` in `src/utils/quality-router.ts` (+224 lines).

Bash command shape:

```bash
claude -p "$(cat <prompt-file>)" \
  --agent ccg/phase-runner \
  --output-format stream-json --include-partial-messages --verbose \
  --max-budget-usd 2.0 \
  --dangerously-skip-permissions \
  --add-dir <workdir> \
  > .context/jobs/<job-id>/progress.jsonl 2>&1
```

`autonomous.md` Step 4.2-4.3 was rewritten: stream-json events stream into `.context/jobs/<job-id>/progress.jsonl`; the main thread reads the last `result.result` field from stdout to parse the ≤200 token SUMMARY (semantically identical to v4.4 Agent spawn).

**Action required**: none. `autonomous` opt-in `useDirectBashInvocation: true` triggers the new path; default off keeps v4.4 Agent spawn semantics.

### 2. Phase 2 (P1b, commit `20fb5fe`): process supervisor + recovery

Treats codex C2 + C4 (process leak / orphan detection).

- **`src/utils/process-tree.ts`** (new): Windows `taskkill /T /F` + POSIX `setsid` process group + signal handler + reconciler (scans `.context/jobs/*/state.json` for stale `running` jobs on session start)
- **`templates/scripts/ccg-phase-runner-launcher.mjs`** (new): wraps `claude -p`, pre-allocates job-id, atomic-writes initial state (parent_pid / cli_pid / process_group_id / cwd / started_at / cmd), monitors exit code, atomic-writes terminal state
- **`src/utils/jobs.ts`**: state.json / result.md / cancel.flag now atomic (temp + rename), no half-written JSON on crash
- **`templates/commands/cancel.md`**: cooperative → cooperative + grace + kill-tree fallback
  - Windows: cancel.flag write → 5s grace → Job Object close kills process tree (incl. nested plugin)
  - POSIX: kill -TERM process group → 5s grace → kill -KILL
- **`templates/hooks/ccg-session-state.cjs`**: SessionStart reconciler scans for stale jobs, PID alive check, matches process start time

**Action required**: none — automatic on `ccg init` / `ccg update`. Existing `.context/jobs/` state files are forward-compatible (extra fields ignored by older readers).

### 3. Phase 3 (P1c, commit `1086aca`) — G2 GATE PASS

Treats codex C1 (memory growth).

Pilot 2/4 matrices (trivial-single N=3 / plugin-single N=2; 4-outer-concurrent deferred via cost guardrail). RSS slope:

| Matrix | N | Total ΔRSS | Avg slope (MB/nested) | Marginal post-warmup |
|--------|---|------------|------------------------|----------------------|
| trivial-single | 3 | 233 MB | 78 | ~7.5 |
| plugin-single (codex) | 2 | 233 MB | 117 | ~5 |

**Critical insight**: codex C1's 200-333 MB/nested linear extrapolation was **invalidated by real measurement**. Real behaviour is warmup-dominant (~210-220 MB first spawn) + tiny marginal (5-15 MB subsequent). 4-outer concurrent worst case ≈ 1.1 GB (vs codex 4-6.7 GB estimate).

`MAX_NESTED_PER_PHASE = 3` enforced in `quality-router.ts`. `degraded.flag` written when RSS > 4GB triggers phase-runner self-implementation fallback.

**G2 PASS** → Phase 6 nested G-plan fully enabled.

### 4. Phase 4 (P1d, commit `285b2ac`) — G3 GATE PASS

Treats codex C3 (broker.log race under 20-way concurrency).

- **`src/utils/broker-log.ts`** (new): writer + reader + 8-field strong schema (tx_id via `crypto.randomUUID`); `getTxLineage(txId)` helper for correlation
- **launcher**: injects `CCG_BROKER_TX_ID` / `CCG_BROKER_LOG_PATH` / `CCG_OUTER_CLI_PID` / `CCG_JOB_ID` into spawned CLI env; persists `broker_tx_id` in state.json

Stress test results:

| Test | Scale | Wall | Result |
|------|-------|------|--------|
| tx_id uniqueness | 100,000 spawn | 227ms | **0 collisions** |
| Concurrent isolation | 4 outer × 5 nested × 100 iter = 2000 spawns | 79s | **0 cross-tx misattribution** / **0 inconsistent terminals** |

**G3 PASS** → Phase 6 may enable nested plugin spawn safely.

**Known minor**: cross-platform stress ran on a single machine (Windows). Cross Windows + Linux consistency is left to Phase 8 dogfood / user validation.

### 5. Phase 5 (P1e, commit `c722d08`)

Treats codex C5 (cost estimation accuracy).

`tests/poc/prompt-cache-bench.ts` + 2 repos (ccg-workflow heavy CLAUDE.md / minimal) × rapid TTL × 5 spawn = 10 real claude CLI subprocess data points.

| Repo | min | mean | p50 | p90 | max | wall p50 |
|------|-----|------|-----|-----|-----|----------|
| ccg-workflow (heavy CLAUDE.md ~46k) | $0.302 | $0.374 | $0.308 | **$0.473** | $0.473 | 22.7s |
| minimal (empty cwd) | $0.024 | $0.115 | $0.175 | $0.175 | $0.175 | 13.2s |

**D3 budget tier verdict**: worst-case p90 $0.473 × 7.5 spawn/phase ≈ $3.55 < debate floor $5. **No revision needed** (fast=$1 / triple=$2 / debate=$5).

**Warm cache critical finding**: minimal cell idx 0-2 cost $0.175 each (cache_create=26,283); idx 3-4 dropped to **$0.024** (cache_create=0, cache_read=46,967) — **86% reduction**. Validates PoC T3 cold→warm 27× projection.

Autonomous 8-phase milestone real-cost projection:

| Tier | Cold cache | Warm cache (×0.5) |
|------|------------|-------------------|
| fast | $7-12 | $4-7 |
| triple | $15-27 | $10-15 |
| debate | $22-40 | $15-22 |

**G4 PASS** (p90 $0.473 << $30/run threshold).

### 6. Phase 6 (P1f, commit `097cda7`): nested G-plan opt-in + launcher wiring

Treats two responsibilities:

(a) **Engine constraint correction**: v4.0.1 "⚠️ 引擎层硬约束" section in `templates/commands/agents/phase-runner.md` is **removed**. That constraint stated "subagent nested spawn is engine-blocked" — true only for in-process sidechain mode. In v4.5 CLI subprocess mode, T9 PoC shows the spawned `claude -p` process can use `Agent` tool natively. The section is replaced by a new "Nested rescue delegation (v4.5 P1f opt-in)" section documenting trigger condition, type routing matrix (backend → codex / frontend → gemini), CAP=3 rule, supervisor degraded.flag protocol, plugin spawn failure fallback.

(b) **Launcher wiring**: Phase 2 shipped `ccg-phase-runner-launcher.mjs` but Phase 1's `autonomous.md` spawn snippet bypassed it. Phase 6 wires it in:

- `quality-router.ts`: `BuildPhaseRunnerLauncherOptions` interface + `buildPhaseRunnerLauncherCommand(phase, options)` helper outputs `node launcher.mjs --job-id <id> --prompt-file <path> -- claude -p ...`
- `autonomous.md` Step 4.2-4.3: spawn snippet rewritten to `Bash(node launcher.mjs ...)` — unlocks Phase 2 supervisor capabilities (atomic state, reconciler, kill-tree)
- `quality-router.ts`: `PhaseMeta.nestedRescue?: boolean` field + `parseNestedFlag(args)` helper + `resolveNestedRescue(input)` (priority: phase frontmatter > CLI flag > default false)
- `status.md` Cancel mode E step 5: cleared `[v4.5-p2-pending]` tag, now invokes `process-tree.ts` `killProcessTree({pid, pgid, graceMs})`

**Default `--nested=off` + `useLauncherWiring=false`**: 100% equivalent to v4.5 v1 (commit `285b2ac` baseline). Single test §7 in `nestedGplan.test.ts` asserts plan structure is byte-for-byte identical.

**Action required**: none. To opt into nested G-plan:

```bash
/ccg:autonomous --nested=on
# OR per-phase: roadmap.md frontmatter `nested_rescue: true`
```

### 7. Phase 7 (P2, commit `614d742`): `/ccg:status` v2

Treats gemini U1 + U3 UX (loss of sidechain inline UI).

- **`templates/commands/status.md`** extended to dashboard + tail dual mode
- **`src/utils/stream-renderer.ts`** (new, 305 lines): `renderEvent / renderJsonl / progressBar / formatElapsed / shouldRenderEvent / parseStreamLine / formatLine / truncate`. Event filter — drops `system/init`, `content_block_delta`, `stream_event/message_*`; keeps `tool_use`, `hook_started`, `assistant` (summary), `rate_limit_event`, `result/success`
- **`src/utils/stuck-detector.ts`** (new, 274 lines): 3 detection rules — loop (sha1 args hash, 3 same calls), slow-tool (30s default), stalled (5min default)
- **ASCII-7 progress bar regex** enforced via `/^[=> ]+$/` test → Windows cp936 / cmd.exe / PowerShell safe

**Usage**:

```bash
/ccg:status                  # dashboard mode — multi-phase aggregated mini progress
/ccg:status --tail <job-id>  # single-line overwrite live progress + stuck warnings
/ccg:status --cancel <phase-id>  # cooperative cancel + kill-tree fallback
```

## Migration steps (recommended, 5 minutes)

After upgrading to v4.5.0 via `npx ccg-workflow update`:

### Step 1: Verify launcher install

```bash
ls ~/.claude/scripts/ccg-phase-runner-launcher.mjs
# Should exist, ~2.5 KB
```

### Step 2: Verify state directory permissions

```bash
mkdir -p .context/jobs
# Must be writable by your user — `claude -p` will fail to start if not
```

### Step 3: Try a small autonomous run with `--nested=off` (default)

```bash
/ccg:autonomous --quality=fast --nested=off
# Should complete with v4.5 launcher path (atomic state writes, kill-tree on cancel)
# Verify .context/jobs/<job-id>/state.json exists with broker_tx_id field
```

### Step 4: Try `--nested=on` with a non-critical phase

Pick a backend phase that would benefit from codex rescue. Watch RSS via Task Manager (Windows) / `top` (POSIX). Expected:

- Marginal RSS growth ~5-15 MB per nested spawn after first warmup
- `MAX_NESTED_PER_PHASE = 3` enforced (4th spawn rejected with degraded.flag)
- `claude.exe` orchestrator RSS stays < 1 GB throughout

### Step 5: Try `/ccg:status --tail` on a running job

```bash
# In one terminal:
/ccg:autonomous --quality=triple

# In another terminal:
/ccg:status --tail <job-id>
# Single-line live progress with cp936-safe ASCII-7 bar
# Stuck warnings appear on tool_call loops / slow tools / stream stall
```

## Known unverified items

### 1. uni-iam real-environment RSS validation (Release entry criteria)

The v4.5 design target is `claude.exe` RSS < 8GB throughout a 5+ phase autonomous run on the uni-iam project (v4.4.x same workload measured 23GB).

**Why deferred**: chicken-and-egg — the v4.5 Phase 1-7 dogfood inside ccg-workflow itself ran on the v4.4.3 spawn path (because Phase 1's mechanism only takes effect on **install** + **new session**, not retroactively in the same session that built it). Cross-project validation also requires uni-iam access which the dogfood did not have.

**How to validate** (user action):

```bash
# Step 1: cd into uni-iam project
cd /path/to/uni-iam

# Step 2: install latest CCG
npx ccg-workflow update

# Step 3: open NEW Claude Code session (not /clear — full restart)
# Step 4: run a 5+ phase milestone
/ccg:autonomous --quality=triple

# Step 5: monitor RSS
# Windows: Task Manager → claude.exe → Memory column
# POSIX: top -p $(pgrep -f "claude") -d 5

# Expected: claude.exe RSS stays < 8GB throughout
# If hit 8GB: file an issue with .context/jobs/*/state.json + RSS trace
```

### 2. Cross-platform broker.log consistency

Phase 4 stress (100k uniqueness + 2k concurrent) ran on a single Windows machine. Cross Windows + Linux consistency is implicit from `crypto.randomUUID` semantics but not directly stress-tested.

**How to validate**: run `tests/stress/broker-concurrent.ts` on a Linux box and compare results.

### 3. debate / impl path silent fallback

v4.4.3 schema validation catches violations after-the-fact but cannot prevent them at runtime. Full elimination requires the openclaw route (debate also Bash-direct), which is token-budget prohibitive in v4.5. Tracked as v4.6+ candidate.

## Rollback path

If v4.5 causes issues in your workload:

| Symptom | Quick mitigation | Full rollback |
|---------|------------------|---------------|
| RSS leak persists despite v4.5 | Set `--nested=off` (default) | `npm install -g ccg-workflow@4.4.3` + restart session |
| Bash spawn fails > 10% rate (G1 trigger) | Set autonomous `useDirectBashInvocation: false` (env var or per-call flag) | Same as above |
| Nested cap too aggressive | Override per-phase in roadmap.md frontmatter `nested_rescue: false` | — |
| Launcher reconciler killing live jobs | Delete `.context/jobs/*/state.json` (clears stale state) | — |

The four built-in decision gates (G1-G4) provide automatic degradation paths — see `.ccg/v4.5-roadmap.md` "Decision Gate" table.

## Plugin known issues + workarounds

CCG 调 codex / gemini 走 plugin（v4.4+），plugin 自身 bug 在用户机本地 patch 才能立刻缓解。**已知 issue + 修补步骤维护在**: [`PLUGIN-PATCHES.md`](./PLUGIN-PATCHES.md)。

当前清单（v4.5.1 起记录）：

| ID | Plugin | 症状 | 状态 |
|----|--------|------|------|
| P-1 | `gemini@google-gemini` v1.0.1 (Windows) | `spawnBackgroundWorker` (gemini-companion.mjs:701) 漏 `windowsHide: true` → spawn 抢焦点 | 本地 patch 可缓解 |
| P-2 | 同上 (a/b 两处) | 底层 `spawn("gemini", ["--acp"])` 缺 `shell:true + windowsHide` → ENOENT + 闪 cmd 黑窗（acp-broker.mjs:85 + lib/acp-client.mjs:243）| 本地 patch 可缓解 |
| P-4 | 同上 | broker daemon `spawn("node", ["serve", ...])` (lib/broker-lifecycle.mjs:137) 缺 `windowsHide` → 启动闪框 | 本地 patch 可缓解 |
| P-5 | 同上 | `runCommand` spawnSync (lib/process.mjs:18) 缺 `windowsHide` → `gemini --version` / git 命令闪框 | 本地 patch 可缓解 |
| P-6 | 同上 | `taskkill` spawnSync (lib/process.mjs:103) 缺 `windowsHide` → cancel 时闪框 | 本地 patch 可缓解 |
| P-7 | 同上 | `spawnDetached` 公共 helper (lib/process.mjs:134) 缺 `windowsHide` → 间歇性闪框 | 本地 patch 可缓解 |

**总共 7 处 spawn 点漏写 `windowsHide: true`**。一键修复：`node ~/.claude/.ccg/scripts/repatch-gemini-plugin.mjs`（CCG v4.5.1+ 装好）。Patch 后必须重启 broker daemon 才能生效（详见 [PLUGIN-PATCHES.md](./PLUGIN-PATCHES.md)）。

**重要**：每次 `claude plugin update <name>` 会覆盖 cache 目录，所有手动 patch 必须重补。
