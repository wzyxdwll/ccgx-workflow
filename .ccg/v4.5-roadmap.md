# CCG v4.5 Roadmap — phase-runner Bash subprocess + 三层 OS 进程隔离

**Status**: Draft (2026-05-06) — pending user approval
**Verdict source**: `.ccg/poc-v45/poc-results.md` (PoC) + `codex-review.md` + `gemini-review.md`
**Route**: 激进路线（含 nested G-plan opt-in）
**Estimated wall time**: 12-15 person-days

---

## Milestone 总目标

把 `Agent(subagent_type="phase-runner")` 主进程内 sidechain spawn 替换为 `Bash(claude -p --agent ccg/phase-runner)` OS-level 子进程。三层进程隔离全部硬边界：

```
主线 claude.exe (orchestrator)
  └─ Bash(claude -p --agent phase-runner)         L1 OS boundary
       └─ CLI 子进程 (phase 工作内存)
            └─ Agent(codex:codex-rescue)          L2 OS boundary (P1f only)
                 └─ plugin 进程 (代码改动沙箱)
```

L1 治 v4.4.x 实测的主进程 RSS leak（23GB / 7.5h 把 VM 弄崩）。L2 解锁原 v4.0 G 方案设计意图（多模型 nested 协作），首次在 CCG 历史上生效。

## Phase 列表（7 个 sub-phase + 1 release phase）

### P1a — 外层 CLI subprocess MVP

**Type**: backend
**Quality**: triple
**Depends on**: 无
**Wall**: 3-4 d

**Goal**: phase-runner 从 `Agent(...)` 改为 `Bash(claude -p --agent ccg/phase-runner ...)`，nested 默认关闭，phase-runner 子进程内自己 Read/Write/Edit/Bash 改代码（保持 v4.0 dogfood 行为）。

**Files**:
- `src/utils/quality-router.ts`：扩展 `useDirectBashInvocation` 选项推广到 impl wave
- `src/utils/quality-router.ts`：新增 `buildPhaseRunnerBashCommand(phase, brief, jobId)` helper
- `src/utils/wave-scheduler.ts`：phase-runner 类型 spawn 走 bashCommand 路径
- `templates/commands/autonomous.md`：spawn 段落改写（Step 4.2-4.3）
- `templates/commands/agents/phase-runner.md`：**不动**（直接 `--agent` 加载）

**生成的 Bash 命令样板**（来自 PoC D1-D8）：
```bash
claude -p "$(cat <prompt-file>)" \
  --agent ccg/phase-runner \
  --output-format stream-json \
  --include-partial-messages \
  --verbose \
  --max-budget-usd 2.0 \
  --dangerously-skip-permissions \
  --add-dir <workdir> \
  > .context/jobs/<job-id>/progress.jsonl 2>&1
```

**Acceptance**:
- [ ] phase-runner spawn 100% 走 Bash 路径，模板渲染零 Agent tool 调用
- [ ] stream-json 流式输出落 `.context/jobs/<job-id>/progress.jsonl`
- [ ] final result 从 stdout 末行 `result.result` 字段 parse 出 SUMMARY 字符串
- [ ] 主线 ≤200 token 摘要回流（与 Agent spawn 行为对齐）
- [ ] dogfood 跑 1 个 milestone（5+ phase）端到端通过
- [ ] 单测覆盖：bashCommand 生成、参数转义、Windows 路径、stream parsing

**Risk**:
- Cold start 实测 vs PoC 估算偏差大 → 跑 P1e benchmark 时校准
- `--verbose` + `stream-json` 行为变化 → 多版本 Claude CLI 测试

### P1b — Process supervisor + recovery

**Type**: backend
**Quality**: triple
**Depends on**: P1a
**Wall**: 2-3 d

**Goal**: 解决 codex C2+C4。所有 v4.5 子进程必须由 supervisor 管理：原子状态写、PID 跟踪、Job Object（Windows）/ 进程组（POSIX）、startup reconciliation、kill-tree。

**Files**:
- `src/utils/jobs.ts`：现有 `state.json/result.md/cancel.flag` 改为 temp-file + rename 原子提交
- `templates/scripts/ccg-phase-runner-launcher.mjs`（新建）：包装 `claude -p` 调用，预先 alloc job-id、写 initial state（含 parent_pid / cli_pid / process_group_id / cwd / started_at / cmd）、监控 exit code、原子写 terminal state
- `src/utils/process-tree.ts`（新建）：Windows Job Object 创建（参照 `codeagent-wrapper` Go 代码） + POSIX setsid + signal handler + reconciler
- `templates/commands/cancel.md`：从 cooperative 升级为 cooperative + grace + kill-tree fallback
- `templates/hooks/ccg-session-state.cjs`：SessionStart 加 reconciler，扫 `.context/jobs/*/state.json` 检查 stale running

**Acceptance**:
- [ ] 13 种崩溃路径（codex C2 表）每种 fault-injection 测试 + 通过
- [ ] Windows: 主线 Ctrl+C → cancel.flag 写 → 5s grace → Job Object close 杀进程树（含 nested plugin 进程）
- [ ] POSIX: kill -TERM 进程组 → 5s grace → kill -KILL
- [ ] 启动 reconciler：scan stale running jobs，PID alive 检查，匹配进程 start time
- [ ] state.json 永远 atomic-commit（temp + rename），任何进程崩溃不留半截 JSON
- [ ] CLI 子进程崩溃时主线收到 exit code，转 `state=failed`
- [ ] orphan plugin 进程在主线退出后被 reconciler 下次启动时清理

**Risk**:
- Windows Job Object API 差异（Win10 vs Win11）→ 复用 `codeagent-wrapper` 现有 Go 实现做 reference
- nested plugin 是否真的继承 Job Object → P1d stress test 验证

### P1c — Memory stress gate（nested 上限决策）

**Type**: backend
**Quality**: fast
**Depends on**: P1b
**Wall**: 1 d

**Goal**: 解决 codex C1。在真 CLI 子进程内跑 5/10/20 次 nested Agent spawn，测 RSS 累积斜率，决定 nested 默认上限（P1f 用）。

**Tests**:
- `tests/poc/nested-rss-stress.ts`（新建）：
  1. 启动 1 个 phase-runner CLI 子进程，记 baseline Working Set
  2. 内部连续 spawn 5 个 `general-purpose` agent，记每次 RSS
  3. 重复 10 / 20 次
  4. 用真 plugin agent (`codex:codex-rescue` / `gemini:gemini-rescue`) 重跑全套
  5. 多 outer CLI 并发：4 outer × 5 nested = 20 路并发 RSS 监测
- 输出 `.ccg/poc-v45/nested-rss-bench.md`

**Acceptance**:
- [ ] RSS 数据 4 张表（trivial/plugin × single-outer/4-outer-concurrent）
- [ ] 测出 per-nested-spawn retained slope（MB/spawn）
- [ ] 决定 default cap：`max_nested_per_phase` （建议保守值：3）
- [ ] CLI 子进程 RSS > 4GB 时 supervisor 主动降级（写 `degraded.flag` 让 phase-runner 切自实施模式）
- [ ] 实测斜率 > 500MB/nested 时直接 NO-GO P1f（回头优化）

**Decision gate**: P1c 出数据 → 决策 P1f 是否能上线。如果 worst case 超阈值，P1f 回头改设计或推迟到 v4.6。

### P1d — Broker transaction isolation + 20-way stress

**Type**: backend
**Quality**: triple
**Depends on**: P1b（可与 P1c 并行）
**Wall**: 1-2 d

**Goal**: 解决 codex C3。v4.4.2 已识别 broker.log 并发 race hazard，nested G-plan 把并发倍增到 20 路。tx_id 必须 128-bit 唯一 + 严格 correlation。

**Files**:
- `src/utils/broker-log.ts`（新建）：tx_id 生成（crypto.randomUUID），事件 schema 强约束（`tx_id`/`job_id`/`phase_id`/`outer_cli_pid`/`plugin_pid`/`event_type`/`timestamp`/`sequence`）
- 替换 `templates/hooks/ccg-context-monitor.js` 等可能消费 broker.log 的逻辑：只通过 tx_id 关联，禁止 tail-position / time-window / nearest-error 推断
- `templates/scripts/ccg-phase-runner-launcher.mjs`：注入 tx_id 到子进程 env

**Tests**:
- `tests/stress/broker-concurrent.ts`（新建）：4 outer × 5 nested × 随机 sleep × 强制 fail，断言：
  - tx_id 100% 唯一（10 万次 spawn 无碰撞）
  - 每个 tx 的 terminal status 与自己进程结果一致
  - 没有 tx 观察到其它 tx 的 failure/success
  - 跨 Windows + Linux 一致

**Acceptance**:
- [ ] tx_id 唯一性 stress test 通过
- [ ] broker.log schema validation（parser 拒收缺字段事件）
- [ ] 20-way 并发下 0 misattribution
- [ ] 现有 PostToolUse hook 路径（如果有消费 broker.log 的）全部改 tx_id 关联

### P1e — Cost/cache 真实 workdir benchmark

**Type**: backend
**Quality**: fast
**Depends on**: P1a
**Wall**: 1 d

**Goal**: 解决 codex C5。PoC cost 估算 ($2.50/run) 偏乐观 4-13 倍，必须在真实 workdir benchmark 才能给出 budget defaults 和用户预期。

**Tests**:
- `tests/poc/prompt-cache-bench.ts`（新建）：
  1. 在 ccg-workflow 自身（重 CLAUDE.md repo）跑 80 次 CLI subprocess spawn，模拟 triple 档 20-phase autonomous
  2. 在 uni-iam（典型业务 repo）跑同样 80 次
  3. 在最小 repo（CLAUDE.md ≤2k）跑同样 80 次
  4. 每次记 `cache_creation_input_tokens` / `cache_read_input_tokens` / total_cost / model / cwd / prompt_hash / elapsed
  5. 测试 TTL 边界：相邻 spawn (<5min) vs 间隔 spawn (>5min)
  6. 输出 `.ccg/poc-v45/cost-cache-bench.md`，含 p50/p90 cost 表

**Acceptance**:
- [ ] 三类 repo × 两种 TTL 模式 = 6 张数据表
- [ ] 给出 v4.5 默认 `--max-budget-usd` 三档：fast=$1 / triple=$5 / debate=$15（基于 p90 + 50% 余量）
- [ ] 文档化：用户应该期待的 autonomous run cost 范围
- [ ] 如果 p90 > $30/run（业务 repo），触发 `--bare` 中间方案优先级提升

### P1f — Nested G-plan opt-in 渐进开启

**Type**: backend
**Quality**: triple
**Depends on**: P1c + P1d（必须同时通过 gate）
**Wall**: 2 d

**Goal**: P1c+P1d 验证通过后，把 nested G-plan 变成用户可用 feature。默认关闭，opt-in 启用。

**Files**:

  - 默认行为：自实施（与 v4.0 保持兼容）
  - 当 prompt 含 `enable_nested_rescue: true` 时，phase-runner 内部按 phase Type 字段 spawn `codex:codex-rescue` (backend) / `gemini:gemini-rescue` (frontend)
- `src/utils/quality-router.ts`：phase frontmatter 加 `nested_rescue: true|false` override；主线编排器加 global flag `--nested=on|off`
- `templates/commands/autonomous.md`：Step 4.0 加 nested mode 检测 + 注入 phase-runner prompt
- `templates/commands/agents/phase-runner.md`：nested 失败的降级路径（plugin spawn 失败 → 切自实施，标 `degraded`）

**Acceptance**:
- [ ] 默认 `--nested=off` 行为与 v4.5 v1（保守路线）100% 等价
- [ ] `--nested=on` 端到端 dogfood：1 个 frontend phase + 1 个 backend phase
- [ ] nested cap 来自 P1c 测出的 slope（默认 3，可 override）
- [ ] supervisor 监控 CLI 子进程 RSS，超 4GB 自动降级 nested → 自实施
- [ ] broker.log 在 nested 模式下 tx_id 关联正确（P1d 验证）
- [ ] 文档化 `--nested` flag 使用场景 + 风险

**Decision gate**: 如果 P1c worst case 不达标 OR P1d stress 不通过 → P1f 推迟 v4.6，v4.5 v1 走保守路线发布。

### P2 — Status command UX (`/ccg:status` v2)

**Type**: frontend (CLI UX)
**Quality**: triple
**Depends on**: P1a（可与 P1b/P1c/P1d/P1e 并行）
**Wall**: 2 d

**Goal**: 解决 gemini U1+U3。失去 sidechain inline UI 后，`/ccg:status` 必须能复刻"长跑用户的微观干预"能力（早上查岗 / 死循环 debug / 单 phase cancel / 实时 tool call）。

**Files**:
- `templates/commands/status.md`：扩展为 dashboard + tail 双模式
- `src/utils/stream-renderer.ts`（新建）：解析 stream-json 事件 → 用户可读单行
- `src/utils/stuck-detector.ts`（新建）：连续 N 次相似 tool_call 检测 / 单 tool >30s 警告

**UX 规范**（gemini U3）：

Dashboard 模式（`/ccg:status` 无 --tail）：
```
[JOB: ccg-nightly-run]
Phase 1 (Setup)     [====================] 100%  (4m 12s)  ✅
Phase 2 (Refactor)  [==========>         ]  50%  (2m 35s)  🤖 codex-rescue
Phase 3 (Tests)     [                    ]   0%  (queued)
```

Tail 模式（`/ccg:status --tail <job-id>`）：
```
[Phase 2] 08:15:22 🤖 Analyzing auth logic...
[Phase 2] 08:15:30 🛠️  Running tool: read_file (src/auth/oauth.ts)
[Phase 2] 08:15:32 ⏳  Waiting for tool result...
[Phase 2] 08:16:05 ⚠️  Tool taking longer than usual (33s)
```

**Event filter**:
- 丢弃：`system/init`, `content_block_delta`, `stream_event/message_start|stop|delta`
- 保留：`tool_use`, `hook_started`, `assistant`(总结), `rate_limit_event`, `result/success`

**Acceptance**:
- [ ] Dashboard 模式聚合多 phase 进度
- [ ] Tail 模式单行覆写实时 progress
- [ ] 卡点检测触发警告（连续 3 次相同 tool_call / 单 tool >30s）
- [ ] `--cancel <phase-id>` 单 phase cancel（复用 P1b cancel.flag + kill-tree）
- [ ] 跨平台测试（Windows cmd / PowerShell / Linux bash）

## 关键风险与降级路径

| 风险 | 触发条件 | 降级路径 |
|---|---|---|
| P1a Bash spawn 路径有未知 bug | dogfood 失败率 > 10% | 回退到 v4.4.3 Agent spawn + 加 SessionStart RSS 警告 hook（治标）|
| P1c 测出 nested 累积太凶 | per-nested slope > 500MB | P1f 推迟 v4.6，v4.5 v1 仅外层隔离 |
| P1d broker race 难解 | tx_id 唯一性测试失败 | nested 模式禁用 plugin spawn，phase-runner 内部只用 Bash 直调 plugin script（v4.4.2 verify wave 路径） |
| P1e cost p90 超 $30/run | 业务 repo benchmark | 强制提升 v4.5 v2 优先级，install-time 提示用户配 apiKeyHelper |
| Windows Job Object 不靠谱 | 杀父进程 plugin 仍 orphan | 回退 cooperative cancel + 用户手动清理 + 文档警告 |
