## v4.1 Phase 列表（按价值 / 工时排序）

### v4.1-α（高 ROI 集中处理）— 5 days

#### Phase 1：SessionStart hook + 项目记忆自动注入（A4，用户 Q6）

- **Goal**：新会话启动时 Claude Code **自动 Read** `.ccg/roadmap.md` / `.ccg/state.md` / 当前 active phase 的 `.context/<phase>/SUMMARY.md`，注入精简摘要到主线 context
- **机制**：写 `templates/hooks/ccg-session-state.js`（参考 GSD `gsd-session-state.sh`），settings.json hooks 段加 `SessionStart` 注册
- **Acceptance**：
  - 新会话启动时 statusline 显示当前 phase / next action（已部分实现）
  - PostStart 注入 `additionalContext` 含 ROADMAP 头部 20 行 + 当前 phase 状态
  - 单测：mock SessionStart 输入，验证 hook 输出格式
- **工时**：1 天
- **Type**：backend

#### Phase 2：autonomous wave 并行调度（A1, B4，用户 Q2）

- **Goal**：autonomous 命令加 `--parallel [--max-concurrent N]` flag，按依赖图拓扑分波，波内并行 spawn phase-runner
- **机制**：
  - 解析 `.ccg/roadmap.md` 每 phase 的 `Depends on` 字段，建依赖图
  - Kahn 算法分波（无依赖 → wave 1，依赖 wave 1 → wave 2...）
  - 波内并行 spawn 多个 phase-runner（Claude Code 原生支持）
  - 主线管多个 task notification（completion 时进 4.4 推进）
  - 跨 wave 依赖断裂自动 cascade skip
- **Acceptance**：
  - `templates/commands/autonomous.md` 加 Step 4.0 拓扑排序 + `--parallel` 路径
  - 单测：构造 12 phase 依赖图，验证波划分正确（roadmap.md 当前依赖图：Wave 1 = 1, 3, 4, 7, 8, 10, 11；Wave 2 = 2, 5, 6 等）
  - 实测：用 v4.1 自身 dogfood 跑（如果有未完成 phase），验证墙钟时间压缩 30-40%
- **工时**：2 天
- **Type**：backend

#### Phase 3：specialist matrix 路由（A2 改进 A，多模型生硬主修复）

- **Goal**：从"按文件类型分"（前/后端）升级到"按角色 × 层"二维路由
- **机制**：6 个核心命令（plan / analyze / debug / review / optimize / test）加 `--role=architect|critic|implementer|tester|writer` flag
- **路由矩阵**：

  ```
  | Role × Layer  | architect      | critic              | implementer | tester        | writer |
  |---------------|----------------|---------------------|-------------|---------------|--------|
  | backend       | codex (xhigh)  | codex (adversarial) | codex       | codex (tester)| claude |
  | frontend      | gemini         | gemini (adversarial)| gemini      | gemini        | gemini |
  | fullstack     | codex+gemini   | both debate         | runner 决   | runner 决     | claude |
  ```

- **Acceptance**：
  - 6 个核心命令模板含 role flag 解析
  - `templates/prompts/{codex,gemini}/` 已有 6 角色 prompt 库（v3.0 就位）—— 直接复用
  - 单测：mock 命令 input + role，验证选对 prompt 文件
- **工时**：2 天
- **Type**：backend

### v4.1-β（中 ROI 集成处理）— 4 days

#### Phase 4：dedicated challenger 主线扁平编排（A2 改进 C，引擎约束适配）

- **背景**：原 v4.0 Phase 1.5 设计想在 phase-runner 内 spawn `assumptions-analyzer` 做"内置 critic"。**但 2026-05-04 nested-spawn 测试证伪**——Claude Code 引擎层禁止 subagent spawn `Agent`（commit `a7cdffd`）。改为**主线扁平化编排**。
- **Goal**：critical phase 在 phase-runner（implementer）完成后，由**主线**追加 spawn `assumptions-analyzer` / `nyquist-auditor`，让 challenger 用独立 fresh context 挑战 implementer 的产出，找到 critical 让 implementer 修订一次
- **机制（扁平化）**：
  ```
  主线 spawn phase-runner (implementer)  ← fresh ctx
       ↓ 摘要返回
  if phase.frontmatter.Critical == true:
      主线 spawn assumptions-analyzer / nyquist-auditor (challenger, 看 phase 定义 + 改动文件) ← fresh ctx
           ↓ critical findings
      if critical:
          主线 spawn phase-runner (implementer, 含 critical 反馈) → 修订
      else:
          推进
  else:
      直接推进（无 challenger）
  ```
- **Acceptance**：
  - 改 `templates/commands/autonomous.md` Step 4.4 推进逻辑加 challenger 分支判定
  - roadmap.md schema 加 `Critical: true|false` 字段（默认 false，避免每 phase 都付双 spawn cost）
  - challenger 选择规则：phase_type=backend → assumptions-analyzer / phase_type=frontend → nyquist-auditor / fullstack → 都跑
  - 单测：mock phase 标 Critical=true，验证主线追加 challenger spawn；mock challenger 报 critical findings，验证主线再 spawn implementer 修订
- **工时**：1 天
- **Type**：backend

#### Phase 5：原生 debate 原语（A2 改进 B，主线编排）

- **背景**：同 Phase 4——引擎层禁止 subagent 嵌套，多轮对辩**必须主线编排**。
- **Goal**：新命令 `/ccg:debate <topic>` 由主线管 A↔B 多轮对辩（spawn → 接 200-token 摘要 → spawn → 接摘要，cap 3 轮）
- **机制（主线编排）**：
  ```
  主线: spawn A (general-purpose, prompt=propose) → 接 propose 摘要
  主线: spawn B (general-purpose, prompt=challenge + A.propose) → 接 challenge 摘要
  主线: spawn A (prompt=respond + B.challenge) → 接 respond 摘要
  ...
  cap 3 轮 或 B 自报"无 critical 挑战" 即停
  主线综合 → 最终方案 + 分歧点列表
  ```
- **Acceptance**：
  - 新建 `templates/commands/debate.md`（命令模板，含主线编排状态机）
  - 每轮 spawn 子 agent 返回 ≤200 token 摘要（避免 transcript 污染主线）
  - 单测：mock 多轮 propose/challenge/respond，验证 cap 3 触发停止
  - 单测：mock B 自报"无 critical" 即触发提前停止
- **工时**：3 天
- **Type**：backend

### v4.1-γ（低 ROI 收尾）— 3 days

#### Phase 6：清理残留 + 命令面板瘦身（A5, C4）

- **Goal**：
  - ccg init 加 sync 模式 flag（`--sync` 删 ~/.claude 中已不在新模板的文件）
  - 命令面板 31 → 22（合并 / 移到 skill / 删除）
- **机制**：
  - installer.ts 加 sync 路径
  - team-research / team-plan / team-review 折回 `/ccg:team` 子命令（独立调用率低）
  - extract-learnings / forensics / health / map-codebase 移到 skill（user-invocable: true，不进 commands/）
- **Acceptance**：
  - --sync flag 工作 + 单测验证不删用户自建文件
  - 命令面板从 31 降到 22 ± 2
- **工时**：1.5 天
- **Type**：backend

#### Phase 7：Skill 体系优化（C1, C2, C3, A3）

- **Goal**：
  - **C1**：审计 100+ skill description 总长，超 1% 预算的截短到 ≤80 字符
  - **C2**：domains/ + impeccable/ 重型 skill 加 `context: fork` frontmatter
  - **C3**：`frontend-design` 等加 `paths: "*.tsx,*.vue"` 限定
  - **A3**：impeccable 20 description 翻译成中文
- **Acceptance**：
  - 跑 `du -sh ~/.claude/skills/ccg/**/SKILL.md` 总 description 长度脚本审计
  - skill-registry.ts 单测覆盖新 frontmatter 字段解析
- **工时**：1.5 天
- **Type**：backend

| 风险 | 缓解措施 |

## v4.0 → v4.1 哲学转变

| v4.0 | v4.1 |
|------|------|
| **基础设施落地** — phase-runner / state machine / async triplet | **使用体验精修** — wave 调度 / 项目记忆 / 多模型对辩 / skill 优化 |

| 模板硬编码 | role-based 路由 |

v4.0 是"立柱搭梁"，v4.1 是"墙面装饰"——基础结构不动，体验细节大幅升级。
