## 问题陈述

### Ground Truth（实测证据，非推断）

| Tool | subagent_type 名 | 证据 |
|------|------------------|------|
| `Agent(...)` 工具调用 | `codex:codex-rescue` / `gemini:gemini-rescue`（**双前缀**） | 用户 acms phase 9.x 实测 spawn `gemini:rescue` 错误返回 + 本会话 system 顶部 agent 列表 |
| `Skill(...)` 工具调用 | `codex:rescue` / `gemini:rescue`（**单前缀**） | 本会话 system 顶部 skill 列表 |
| Plugin install dir | `codex@openai-codex` / `gemini@google-gemini` | plugin-detection.ts:88-91 既有正确探测 |

### CCG 全仓现状

`grep -rn "codex:rescue\|gemini:rescue" templates/ src/ tests/` 命中 **195 处 / 40 文件**。其中：

- ❌ **production 路径**：`src/utils/quality-router.ts:187,199,205,217` 直接生成 `agent: 'codex:rescue'` 字符串塞给主线 spawn → **这是用户看到的 phase 9.x 错误的根因**
- ❌ **顶层文档**：`templates/CLAUDE.md` x4 / `phase-runner.md` x3 / `autonomous.md` x10+
- ❌ **测试 fixtures 内部不一致**：`tests/fixtures/ground-truth/agent-summaries.sample.json` 同文件 line 4 用双前缀、line 7+13 用单前缀 → interface-auditor ground-truth 锚定**两边都视为 valid**，等于失能
- ⚠️ **概念反向**：`templates/commands/agents/interface-auditor.md:83` 把"双前缀"举为"不存在的错名"反例，但这正是真名 → v4.3 整套防御机制基于错误前提
- ⚠️ **CHANGELOG 反向**：root CLAUDE.md v4.3.0 段写"假设 codex:codex-rescue（实际 codex:rescue）同型事故"——含义反了，实际从来都是双前缀

### dogfood 没暴露的原因

CHANGELOG 自承："12 phase 全用 phase-runner 自实施"——phase-runner 是 CCG 自家 subagent（注册名 `phase-runner`，无前缀），不走 plugin spawn 路径。所以 v4.0 → v4.4 的 dogfood 从未真正调用过 `Agent(subagent_type="codex:rescue")`，错名长期潜伏。直到用户 acms phase 9.x verify wave 主动调用 → 引爆。

### 我自己的责任

我刚才 `fb40937` commit（debate.md broker 修复）原样复制了 4 处错名 `codex:rescue` / `gemini:rescue`，这次一并修。v4.4 P32 phase-runner ground-truth 强约束（"禁止凭训练记忆做编码决策"）就是为防这个，我没遵守，B 流程中借此修补。

## Bedrock Truths（first-principles）

1. **真名是双前缀** — 实测可证伪可观察
2. **Agent ≠ Skill** — 两个 tool，两个命名空间，CCG 文档没区分
3. **错的是 magic string，不是探测/路由逻辑** — plugin-detection.ts / debate-orchestrator.ts / specialist-router 的算法都对，只是它们处理的字符串值错了
4. **fixture 不一致 = interface-auditor 失能** — 必须先把 ground truth 锚到唯一真名才能让 v4.3 P27 防御真生效
5. **批量替换需要分类** — 不是全部 `codex:rescue` 都要变；skill 引用 + 历史叙述 + 反例文档语义不同
