# Plan: v4.4.2 — verify wave Bash direct invocation

## 任务类型
- [x] 后端（verify-orchestrator.ts + 4 模板）
- [ ] 前端
- [ ] 测试 only

## 问题陈述

CCG verify wave 当前走 `Agent(subagent_type="codex:codex-rescue" | "gemini:gemini-rescue")` spawn plugin 子智能体做 cross-vendor 验证。但 plugin 子智能体的 `model: sonnet` wrapper 在 broker 故障 / gemini 空答 / codex 错误时会**自答**（instruct-tuning 反射），主线收到 looks-normal ≤200 token 摘要——silent contamination。

verify wave 是**最高价值用例**（cross-vendor diversity 决定 advance/revise/escalate），contamination 直接进决策路径无下游兜底。

## Bedrock truths

1. plugin spawn 路径上有 sonnet wrapper（`agents/gemini-rescue.md:4` model: sonnet 实测验证）
2. silent fallback 是 LLM instruct-tuning 反射，不分模型大小（opus 更可怕）
3. Bash 直调 plugin 脚本（`*-companion.mjs task ...`）**绕开 wrapper 但复用 broker**，无 LLM 在中间层
4. PostToolUse hook 影响全局 + 并发漏检 + opt-in 摩擦——不值得
5. impl/autonomous/debate 用例有下游 verify 兜底，contamination 进不到最终交付

## 实施步骤

### Wave 1：helper 层加 `useDirectBashInvocation` 选项

**文件**：`src/utils/verify-orchestrator.ts`

**改动**：
1. `VerifySpawnEntry` 加 `invocationMode?: 'agent' | 'bash-direct'` 字段
2. `planVerifyWave(tier, layer, plugins, options?)` 加 `options.useDirectBashInvocation?: boolean`
3. true 时所有 plugin spawn entry 标 `invocationMode: 'bash-direct'`，附带 `bashCommand: string` 字段（含 `node <plugin-script> task ...` 模板）
4. false / 未传时保留现行行为（agent spawn）

预期 ~50 行新代码 + ~10 行 schema 更新。

### Wave 2：测试覆盖

**文件**：`src/utils/__tests__/verifyOrchestrator.test.ts`

**新增 ~30 行**：
- `useDirectBashInvocation: true` 时返回 entry 含 `invocationMode: 'bash-direct'` + `bashCommand` 含 plugin script 路径
- 默认行为（不传 options）保持 agent spawn entry 不变（向后兼容）
- plugin 缺失时仍走 general-purpose（无论 mode）

### Wave 3：4 个模板切换路径

**文件清单**：

**改动模式**（每模板 ~10-15 行）：

## 风险与缓解

| 风险 | 缓解 |
|------|------|
| Bash 直调 stdout 5-50KB 回灌主线吃 context | verify wave 是 phase 末段（已接近 phase 完成）+ 摘要后存 SUMMARY.md，影响窗口短 |
| plugin 路径含版本号（`codex/1.0.4/`），版本升级路径漂移 | 模板用 glob `~/.claude/plugins/cache/openai-codex/codex/*/scripts/codex-companion.mjs`，主线 Bash 跑 `ls` 解析 |
| Bash exit code 0 + empty stdout 容易漏判 | 模板加显式检查：exit≠0 OR stdout 字节<阈值 → 重试 |
| 主线解析 stdout 比解析摘要更复杂 | --json 输出格式稳定（plugin 自带 schema），主线读 `result.text` 字段即可 |
