## Memory stress summary (Phase 3 — `1086aca`)

| Matrix | N | Total ΔRSS | Avg slope (MB/nested) | Marginal post-warmup |
|--------|---|------------|------------------------|----------------------|
| trivial-single | 3 | 233 MB | 78 | ~7.5 |
| plugin-single (codex) | 2 | 233 MB | 117 | ~5 |

**Critical insight**: codex C1's 200-333 MB/nested linear extrapolation is **invalidated**. Real model is warmup-dominant: ~210-220 MB first spawn (process startup + Node heap), then 5-15 MB marginal per subsequent spawn. 4-outer × 5-nested concurrent worst case ≈ 1.1 GB (vs codex 4-6.7 GB estimate).

**Deferred to user dogfood**: matrices 3+4 (concurrent N=10/N=20). Methodology proven on single-outer pilots; estimated full `--all --n=5` cost ≈ $30-80, ~5 min wall.

## Broker tx_id stress summary (Phase 4 — `285b2ac`)

| Test | Scale | Wall | Result |
|------|-------|------|--------|
| tx_id uniqueness | 100,000 spawn | 227ms | **0 collisions** |
| Concurrent isolation | 4 outer × 5 nested × 100 iter = 2000 spawns | 79s | **0 cross-tx misattribution** / **0 inconsistent terminals** |

**Schema**: 8 mandatory fields, `crypto.randomUUID` minted at launcher start, injected as `CCG_BROKER_TX_ID` env into spawned CLI. `getTxLineage(txId)` reader correlates parent ↔ nested events purely by tx_id (no tail-position / time-window / nearest-error heuristics).

**Known minor**: cross-platform consistency (Windows + Linux) verified by `crypto.randomUUID` semantics but not directly stress-tested cross-platform.

## User dogfood validation required

The v4.5 design relies on a **single key validation that this milestone could not perform**:

### Release entry criteria: uni-iam project RSS validation

**Goal**: confirm `claude.exe` orchestrator RSS stays < 8GB throughout a 5+ phase autonomous run on the uni-iam project (v4.4.x same workload measured 23GB).

**Why this milestone could not validate**:
1. **Chicken-and-egg**: Phase 1's mechanism only takes effect on **install** + **new session**. The dogfood ran the v4.4.3 spawn paths because the new code was being built in the same session.
2. **No uni-iam access**: dogfood happens inside ccg-workflow repo only.

**User action required** (5-10 minutes):

```bash
# Step 1: cd into uni-iam project
cd /path/to/uni-iam

# Step 2: install latest CCG
npx ccg-workflow update

# Step 3: open NEW Claude Code session (full restart, not /clear)

# Step 4: run a 5+ phase milestone
/ccg:autonomous --quality=triple

# Step 5: monitor claude.exe RSS
# Windows: Task Manager → claude.exe → Memory column (refresh every 30s)
# POSIX: top -p $(pgrep -f "claude") -d 5

# Expected: claude.exe RSS stays < 8GB throughout the run
# If hits 8GB: file an issue with .context/jobs/*/state.json + RSS time-series
```

**Failure path (if RSS climbs > 8GB)**:
- Trigger G1 gate manually: roll back to v4.4.3 (`npm install -g ccg-workflow@4.4.3`) + open issue
- Decision gate G1 spec: "Phase 1 dogfood failure rate > 10% → revert + SessionStart RSS warning hook (treat-symptoms)"

### Secondary validation: cross-platform broker.log consistency

`crypto.randomUUID` semantics imply portability, but Phase 4 stress ran on Windows only. Suggested: run `tests/stress/broker-concurrent.ts` on a Linux box and compare metrics.
